## 수상

**우수상 — SSAFY 특화 프로젝트** (삼성 청년 SW 아카데미, 2026)

---

## Overview

DailyLog는 음성 기반 AI 일기 시스템이다.
설계의 핵심 제약은 하나였다: 가능한 한 서버 왕복 없이 ASR, LLM 추론, TTS를 실행하면서도 체감 지연을 최소화하는 것.

핵심 엔지니어링 과제:
- int8 양자화 온디바이스 ASR + 음성 감정 분류 동시 수행
- 슬롯 기반 LLM 에이전트로 자연스러운 대화에서 구조화된 일기 추출
- 탐색-활용 균형을 위한 톰슨 샘플링 기반 활동 추천

---

## Problem

기존 일기 서비스는 텍스트 직접 입력이 전제였다. 이 마찰이 일상 완료율을 낮췄다.
음성 우선 접근은 세 가지 제약을 동시에 만들어냈다:

- 모바일 ASR은 자연스러운 대화를 위해 400ms 이하여야 한다
- LLM 에이전트는 고정된 형식 없이도 구조화된 슬롯을 안정적으로 추출해야 한다
- 추천은 새로운 활동을 탐색하면서도 검증된 활동을 활용하는 균형이 필요하다

세 문제 각각을 독립적으로 푸는 건 어렵지 않다. 하나의 저지연 파이프라인 안에서 동시에 해결하는 것이 과제였다.

---

## Architecture

각 레이어를 병목 기준으로 독립 설계한 세 서브시스템:

```
[음성 입력]
     │
     ▼
[ASR: SenseVoice Small / sherpa-onnx]  ← int8 온디바이스
     │  한국어 전사 + 감정 태그 (7종)
     ▼
[LLM 에이전트: Claude Sonnet / FastAPI] ← GMS Anthropic 프록시
     │  슬롯 추출 (7개) → 일기 자동 생성
     ▼
[TTS: 안드로이드 내장]                  ← 온디바이스, 네트워크 없음
     │
     ▼
[추천 엔진]                             ← 톰슨 샘플링 + pgvector
```

## Subsystem 1 — 온디바이스 ASR 파이프라인

### 모델 선정

SenseVoice Small을 Whisper 계열 모델 대신 선택한 이유는 두 가지다:
- ASR과 음성 감정 인식(SER)을 단일 포워드 패스로 동시 수행
- int8 양자화 후에도 Android ARM에서 안정적으로 동작하는 경량 구조

### 양자화 전략

FP32 모델을 sherpa-onnx의 정적 양자화 파이프라인으로 int8 변환:
- 한국어 구어체 음성 샘플로 캘리브레이션
- Attention 및 FFN 레이어에 가중치 전용 양자화 적용
- 수치 민감 경로의 활성값은 FP32 유지해 정확도 붕괴 방지

결과: FP32 대비 모델 용량 약 4배 감소, 중급 Android 기기에서 OOM 없이 안정 실행.

### 감정 태그 추출

SenseVoice Small은 전사와 동시에 7종 감정 레이블을 출력한다:
HAPPY, SAD, ANGRY, DISGUSTED, FEARFUL, SURPRISED, NEUTRAL

이 감정 신호는 LLM 에이전트의 슬롯 채움 컨텍스트에 직접 주입된다.
모델이 텍스트만으로 감정 상태를 추론해야 하는 부담을 줄이는 설계다.

### 한국어 구어체 키워드 보정

낭독체 학습 ASR 모델은 한국어 구어체를 구조적으로 오인식한다:
- 축약형(어떡해 → 어떻게 해)과 문장 종결 조사
- 비격식 어미(했어, 했지, 했거든) 등 격식체와 다른 표현

빈출 구어체 패턴을 보정하는 규칙 기반 후처리 레이어를 적용했다.
LLM 에이전트에 전달되기 전 ASR 정규화 오류로 인한 슬롯 채움 오류를 사전 차단한다.

### 지연 결과

온디바이스 ASR 추론 약 300ms 달성 — 자연스러운 턴 교환을 위한 체감 임계치 이내.

## Subsystem 2 — 슬롯 기반 LLM 에이전트

### 슬롯 스키마

에이전트는 멀티턴 대화에서 7개의 구조화된 슬롯을 추출한다:

| 슬롯 | 내용 |
|------|------|
| `event` | 오늘 있었던 일 |
| `emotion` | 느낀 감정 |
| `cause` | 그렇게 느낀 이유 |
| `highlight` | 오늘의 하이라이트 |
| `challenge` | 힘들었던 점 |
| `tomorrow` | 내일의 목표나 계획 |
| `reflection` | 전반적인 회고 |

### 대화 관리

매 턴마다 에이전트는:
1. 현재 슬롯 채움 상태를 점검
2. 우선순위가 가장 높은 미채움 슬롯을 식별
3. 반말 페르소나(친구처럼 자연스러운 한국어)로 적절한 후속 질문 생성
4. 응답에서 슬롯 값 추출 및 상태 업데이트

시스템 프롬프트는 일관된 페르소나(친근하고 공감적인 반말)를 유지하면서 구조화된 데이터를 안정적으로 추출해야 하는 상충을 해결해야 했다.
초기 버전은 기계적인 슬롯 수집이거나 추출이 불안정했다. 최종 설계는 추출 로직과 페르소나 톤을 프롬프트 레벨에서 분리했다.

### 일기 자동 생성

7개 슬롯이 모두 채워지면 에이전트가 1인칭 한국어 일기를 합성한다:
- 슬롯 값으로 서사 일관성 확보
- ASR 감정 태그와 텍스트 감정 톤을 모두 반영
- 구조화된 요약이 아닌 자연스러운 한국어 산문으로 작성

### 인프라

- Claude Sonnet: GMS(Google Mobile Services) Anthropic 프록시 경유
- FastAPI 서버: 세션 상태, 슬롯 추적, 프롬프트 조립 담당
- Redis: 세션별 대화 컨텍스트 저장 (TTL 기반 만료)

## Subsystem 3 — 온디바이스 TTS

안드로이드 내장 TTS 엔진을 서버 사이드 음성 합성 대신 선택한 이유는 단 하나다: 지연.

서버 TTS는 전체 네트워크 왕복이 필요하다 — 합성 요청, 오디오 인코딩, 전송, 버퍼링.
중급 Android 기기에서 일반적인 네트워크 환경 기준으로 이 과정이 누적되면 사용자의 응답과 에이전트의 다음 질문 사이에 1.4~2.0초의 침묵이 발생했다.

안드로이드 내장 TTS로 전환하면 네트워크 의존성이 완전히 제거된다:
- 오디오 합성이 로컬, 인-프로세스에서 즉시 수행
- 인코딩·전송 오버헤드 없음
- 응답 지연 50ms 이하로 단축

트레이드오프는 음질이다. 서버 TTS가 더 자연스러운 억양을 만들어낸다.
하지만 음질보다 대화 리듬을 우선했다 — 50ms 응답은 대화처럼 느껴지고, 2초 침묵은 흐름을 끊는다.

## Subsystem 4 — 개인화 추천 엔진

### 설계 목표

하루에 하나, 사용자의 가장 낮은 웰니스 축을 타겟팅하는 활동을 추천한다.
같은 활동을 반복 추천해선 안 되지만(다양성), 검증된 활동을 무시해서도 안 된다(활용).

### 5단계 파이프라인

**1단계 — 웰니스 상태 분석**
LLM이 완성된 일기에서 6축 웰니스 점수를 추출한다:
- 안정감 / 성취감 / 에너지 / 연결감 / 기쁨 / 편안함
점수가 가장 낮은 축이 당일 타겟팅 차원으로 선정된다.

**2단계 — 후보 리콜**
다음 기준으로 필터링한 활동 카탈로그에서 후보 6개를 리콜한다:
- 동일 연령대·성별 코호트
- 타겟 웰니스 축과의 적합도

**3단계 — 다중 신호 스코어링**
각 후보에 6개 독립 신호의 복합 점수를 산출한다:

| 신호 | 설명 |
|------|------|
| 축 적합도 | 타겟 웰니스 축과의 정렬 정도 |
| 코호트 점수 | 동일 연령대·성별 세그먼트 내 인기도 |
| 관심사 점수 | 사용자 선언 관심사와의 유사도 |
| 문맥 점수 | 시간대·상황적 관련성 |
| 유사 사용자 점수 | 유사 감정 패턴 사용자 일기와의 pgvector 코사인 유사도 |
| MAB 점수 | 톰슨 샘플링 기반 탐색 점수 |

**4단계 — 다양성 패널티 및 최종 추천**
최근 7일간 추천된 활동에 재등장 패널티를 적용한다.
최종 점수 = 복합 가중치 점수 − 다양성 패널티 → Top-1 선정

**5단계 — 피드백 학습**
사용자 피드백(완료/스킵)을 활동별·사용자별로 저장한다.
각 활동의 Beta 분포 α/β 파라미터를 업데이트해 이후 MAB 점수에 반영한다.

### 왜 톰슨 샘플링인가

| 알고리즘 | 한계 |
|----------|------|
| 엡실론 그리디 | 탐색률이 고정; 관측 보상 분산에 적응하지 못함 |
| UCB | 보상 분포가 유계라고 가정; 초기에 신뢰도 과추정 |
| 톰슨 샘플링 | 사후 Beta(α, β)에서 샘플링; 데이터 누적에 따라 탐색 강도 자연스럽게 조정 |

톰슨 샘플링은 실제 불확실성에 맞춰 탐색 강도가 적응한다. 새로운 활동에 공정한 기회를 주면서도 시간이 지남에 따라 신뢰도 높은 추천으로 수렴한다.

![톰슨 샘플링 밴딧 스코어링]({{ '/assets/img/projects/daily_log_bandit.png' | relative_url }})

### Kafka 기반 추천 풀 확장

사용자가 체크인에서 행복한 감정(HAPPY)을 보고할 때, 해당 순간의 활동 데이터가 Kafka 토픽에 이벤트로 발행된다.
컨슈머가 이 스트림을 모니터링해 활동이 전체 사용자에서 3회 이상 등장하고 2회 이상 긍정 피드백을 받으면 영구 추천 카탈로그로 승격한다.

수동 큐레이션 없이 실제 사용자 행동 데이터로 추천 풀이 유기적으로 성장하는 피드백 루프다.

### pgvector 유사도 탐색

각 일기를 텍스트 임베딩으로 변환해 pgvector 익스텐션이 활성화된 PostgreSQL에 저장한다.
추천 시점에 사용자의 최신 일기 임베딩으로 전체 사용자 일기를 코사인 유사도 탐색한다.
유사한 감정 패턴의 사용자가 긍정 반응을 보인 활동이 유사 사용자 신호에서 가산점을 받는다.

## Result

| 지표 | 이전 | 이후 |
|------|------|------|
| 일기 작성 시간 | 약 20분 | 약 5분 |
| 감소율 | — | 약 70% |
| ASR 추론 지연 | N/A (서버 측) | 약 300ms (온디바이스) |
| TTS 응답 지연 | 1.4~2.0초 (서버 TTS) | 50ms 이하 (온디바이스) |
| 슬롯 추출 | 수동 입력 | 7개 슬롯 자동 추출 |
| 추천 다양성 | 고정 리스트 | MAB 적응형 |

---

## Key Insight

300ms ASR 목표는 양자화와 모델 선정을 사후 압축이 아닌 공동 설계 결정으로 다뤘기 때문에 달성 가능했다.
LLM 에이전트의 안정성은 더 강력한 모델이 아니라 프롬프트 레벨에서 페르소나와 추출 로직을 분리한 설계에서 나왔다.
추천 시스템의 가치는 5단계 파이프라인 구조에서 나왔다. 톰슨 샘플링 단독으로는 표면적인 개인화에 그쳤을 것이다.

각 서브시스템의 제약이 해법을 정의했다. 최적화 전 측정이 일관된 원칙이었다.
